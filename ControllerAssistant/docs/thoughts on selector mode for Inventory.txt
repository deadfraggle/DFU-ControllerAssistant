That’s promising. The inventory legend proving you can latch onto parentPanel and draw over the window is a big milestone.

And yes: your instinct is right. A real Selector Mode is probably doable, but it is no longer “just input translation.” It becomes a UI model problem:

what is selectable

where it is on screen

how selection moves

how selection maps to the underlying item/action

So the hard part is not drawing the selector box. The hard part is building a reliable understanding of the inventory window’s layout and state.

My thoughts:

This is feasible

What you described is much more realistic than trying to physically move DFU’s pointer.

A software selector is the clean approach:

keep your own selected slot/index

draw your own highlight box

on Action1 / Action2, invoke the same logic the inventory window would use for the hovered/clicked item

That is exactly the kind of problem your mod architecture is starting to be good at.

The project splits into 3 separate problems

Treat these as separate jobs.

1. Visual selector

This is the easiest part now.
You already proved you can overlay UI on inventory. So a hollow box, or even a translucent frame, is the least scary part.

2. Navigation model

This is the real design challenge.

You need to decide what the selector is moving across:

a list of inventory slots

equipment paper-doll regions

side buttons like Equip / Remove / Use

tabs across the top

scroll arrows

wagon side

ingredient side

This is why I would not start by trying to make Selector Mode cover the whole inventory screen.

Start with one constrained region only.

Best first target:

the main inventory item grid

Why that first:

it is rectangular

it visually suggests slot-to-slot movement

it is the place where controller selection helps most

Once that works, later you can expand selector mode to other regions.

3. Action dispatch

Once selected, what does Action1 do?
What does Action2 do?
How do you trigger scroll when selection moves off visible rows?

That depends on how the inventory window internally stores visible items and selected items. This is where reflection or source reading will matter most.

The safest development order

I would strongly recommend:

Phase A

Build a selector box that moves over a fake grid in the inventory item list area.

No item logic yet.
No clicks yet.
Just:

enter selector mode

move left/right/up/down

highlight slots

clamp to visible bounds

This tells you whether the movement feels good.

Phase B

Make the selector understand scrolling.
When moving above the first visible row or below the last visible row:

trigger the existing inventory scroll behavior

keep selector in the same column

This is already a substantial feature.

Phase C

Map the selector to the actual item under that slot.
Now Action1 can act on the selected visible item.

Phase D

Expand beyond the item grid into buttons, tabs, equipment slots, wagon, etc.

If you try to solve all of that at once, the project will balloon fast.

I would avoid this approach

I would not start by trying to model every clickable region in the inventory screen as one giant unified selector graph.

That would be elegant later, but miserable to build first.

Instead:

start with one region

prove the concept

then decide whether to connect regions with explicit transitions

For example:

left side = equipped slots

center/right = inventory grid

top = tabs

button column = vertical menu

Those could eventually be separate selector zones, each with its own rules.

What you’ll likely need from the inventory window

At some point, yes, this becomes a task for someone who understands how the window arranges items.

Specifically, you’ll want to discover:

which field holds the visible item collection

how many rows/columns are visible

where the grid starts on screen

how scroll offset is tracked

what method gets called for left-click equip/use/select

what method gets called for right-click/remove/info behavior

how equipment slots are represented

Your field/method dump tool is perfect for this stage. Not because it tells you the answer directly, but because it narrows the search.

My honest take

This is very achievable, but only if you keep it narrow at first.

The danger is not technical impossibility.
The danger is turning “selector mode for the inventory grid” into “full controller rewrite of the whole inventory menu.”

Those are very different-sized jobs.

My recommendation

Define Selector Mode v1 as only this:

activates in inventory

selector box appears over the visible inventory item grid

D-pad moves between visible slots

moving past top/bottom scrolls

Action1 performs primary item action

Action2 performs alternate/remove action

Legend updates to show selector controls

If you can get that, you’ll already have something impressive and useful.

And yes, based on what you’ve already pulled off with automap legends and inventory overlay attachment, this feels like the right next frontier for the mod.
